Why This Matters

Common metaphors are often limiting:

  • “AI is a search engine” → leads to simple Q&A usage
  • “AI is a calculator” → leads to expecting deterministic answers
  • “AI is a robot” → leads to mechanical, command-based interaction


More expansive metaphors unlock possibility:

  • “AI is a collaborator” → leads to iterative co-creation
  • “AI is a lens” → leads to using it to see your own work differently
  • “AI is a medium” → leads to thinking about new forms of expression


Today we’ll explore AI as a medium for knowledge transmission — a fundamentally new way to share what you’ve learned.

A Working Vocabulary

Before we go further, let’s define some terms you’ll hear today. These are practical terms — the language of working with AI, not the theory behind it.

  • Prompt — The text you give to an AI model. A question, an instruction, a conversation. Prompting well is a skill, not a formula.

  • Context window — How much text the AI can “see” at once. Think of it as working memory. Today’s models can hold roughly the length of a novel.

  • Token — The unit AI models read in. Roughly ¾ of a word. Matters because context windows and pricing are measured in tokens.

  • Model — The AI engine itself (e.g., Claude, GPT, Gemini). Different models have different strengths — like choosing between tools in a workshop.

A Working Vocabulary (cont.)

  • Agent — An AI that can take actions, not just generate text. It can read files, run code, search the web, call APIs — and decide what to do next.

  • Harness — The scaffolding around an AI model that gives it tools, rules, and context. The model is the engine; the harness is the car. (Claude Code, Cursor, and Codex are all harnesses.)

  • API (Application Programming Interface) — A structured way for software to talk to other software. When your dissertation exposes an API, any program (including an AI agent) can query your data.

  • MCP (Model Context Protocol) — A standard that lets AI agents discover and use APIs automatically. Think of it as a universal adapter between AI and your tools.

A Working Vocabulary (cont.)

  • Render — Turning source files into finished output. Quarto renders your .qmd files into a website, PDF, or slides.

  • Repository (repo) — A folder tracked by git. It holds your code, text, data, and their full history. Your dissertation lives in a repo.

  • Fork — Your own copy of someone else’s repo. You can modify it freely without affecting the original. Today you’ll fork the dissertation framework.

  • Monorepo — Multiple related projects in a single repository. The web application uses a monorepo with shared libraries.


You don’t need to memorize these. They’ll become natural as we use them today.

Part I

The Dissertation as Transmission Mechanism

What is a dissertation, really?

A Question Worth Asking


What is a dissertation?


At its core, a dissertation is a knowledge transmission mechanism — a technology for moving understanding from one human mind to another.

Like all technologies, it has evolved. But perhaps not as much as we think.

The Oral Tradition (Pre-1200s)

Knowledge lived in people, not documents.

  • Master-apprentice relationships
  • The disputatio — public oral defense
  • Memory and rhetoric as core scholarly skills
  • Knowledge died with its holders


The transmission was synchronous, ephemeral, and one-to-few.

The Medieval University (1200s–1400s)

The university formalized knowledge transmission:

  • Lectio (reading aloud from texts) → Quaestio (questioning) → Disputatio (debate)
  • The thesis: a proposition to be defended
  • Manuscripts copied by hand — expensive, scarce, fragile
  • Knowledge was institutional, trapped in monastery libraries and university halls


The dissertation emerged as a ritual of certification — proving mastery to a small audience of peers.

The Printing Press (1450s→)

Gutenberg didn’t just print bibles. He transformed knowledge transmission.

  • Dissertations could be reproduced and distributed
  • The bound thesis became a physical artifact — stored on library shelves
  • Knowledge became asynchronous for the first time: write once, read by many
  • But the format stayed the same: linear text, single author, single reader at a time


The printing press changed the reach but not the modality.

The PDF Era (1990s→)

The digital dissertation:

  • ProQuest, institutional repositories, Google Scholar
  • Global reach — anyone with internet can access it
  • But a PDF is a digital photocopy of a printed page
  • Still linear, still text, still one-human-to-one-human
  • Data and code hidden behind “available upon request”


We digitized the container but not the knowledge.

600 Years of Transmission


In every era, the fundamental model has been the same:

One human writes → One human reads

What if that’s no longer the only option?

Part II

New Modes of Knowledge Transmission

APIs and AI change the equation

The Shift

For the first time in history, the consumers of research are not exclusively human.


  • AI agents can read, summarize, and reason about research
  • APIs make data and analyses queryable — not just readable
  • Interactive applications let non-specialists explore findings themselves
  • MCP tools let AI systems discover and invoke your research programmatically

One Source, Many Channels

What APIs Enable

An API doesn’t just serve data — it makes knowledge addressable.


Traditional approach:

“The mean reading score for 2023 was 493. See Table 4.2 on page 87.”

A human must find page 87, locate Table 4.2, and read the value.

API approach:

GET /api/summary?content_area=READING&year=2023
{
  "mean_score": 493.2,
  "median_score": 491.0,
  "n": 4850
}

Any system — human, AI, or application — can query this directly.

What AI Agents Enable

With Model Context Protocol (MCP), AI agents can discover your research:


{
  "name": "dissertation_summary",
  "description": "Get summary statistics from the dissertation's
                  educational assessment dataset",
  "parameters": {
    "content_area": "READING | MATHEMATICS | All",
    "group_by": "GRADE | YEAR | DISTRICT_NUMBER | NONE"
  }
}


An AI reading your dissertation doesn’t just see text — it can invoke your analyses, query your data, and explore your findings interactively.

The Multi-Modal Dissertation

The dissertation is no longer just a document. It’s a platform.

Part III

The AI-Native Dissertation Framework

R Package → API → MCP → NextJS

Architecture Overview

Layer 1: The R Package

The statistical engine — where the actual research lives.

# Core analysis functions
summarizeAssessment(content_area = "READING", group_by = "GRADE")
analyzeAssessment(analysis_type = "achievement_trends")
getChapterSummary(chapter = "methods")
getAvailableAnalyses()


This is a standard R package. devtools::install(), R CMD check, testthat tests, roxygen2 documentation — nothing exotic.

The key difference: it’s designed to be consumed by other systems, not just humans in an R console.

Layer 2: The REST API

R functions exposed as HTTP endpoints via RestRserve:

# In R/api.R
app$add_get("/summary", function(req, res) {
  result <- summarizeAssessment(
    content_area = req$parameters_query[["content_area"]],
    group_by = req$parameters_query[["group_by"]]
  )
  res$set_content_type("application/json")
  res$set_body(jsonlite::toJSON(result, auto_unbox = TRUE))
})


Six endpoints: /summary, /analysis, /chapters, /analyses, /data/variables, /openapi.json

Every R function that produces insight is now addressable via HTTP.

Layer 3: MCP Tools

Model Context Protocol — the bridge to AI agents:

{
  "tools": [
    {
      "name": "dissertation_analysis",
      "description": "Run a typed analysis on the assessment data",
      "inputSchema": {
        "type": "object",
        "properties": {
          "analysis_type": {
            "type": "string",
            "enum": ["achievement_trends", "score_distribution",
                     "district_comparison", "cohort_progress"]
          }
        }
      }
    }
  ]
}

Layer 4: NextJS Frontend

Interactive web application consuming the same data:

  • Data Explorer — filter, group, and visualize assessment data
  • Chapter Navigation — dissertation content with linked analyses
  • Recharts Visualizations — achievement trends, score distributions
  • Dual-mode API client — live R server in development, static JSON in production


// The same analysis, now in a React component
const { data } = useAnalysis('achievement_trends', 'READING');

The Key Insight

You write your analysis once, in R.
It surfaces as an R function, an API endpoint, an MCP tool, a chart, and a table — automatically.


This is what we mean by “leave the tech to the AI.”

Your job is the content — the research questions, the analysis logic, the interpretation. The framework handles distribution.

Step 1: Fork the Repositories

# Fork via GitHub UI, then clone:
git clone https://github.com/YOUR-USERNAME/dissertation-rpkg.git

  https://github.com/YOUR-USERNAME/dissertation-ai.git

cd dissertation-rpkg/dissertation
Rscript -e "devtools::install_deps()"

# Install Node dependencies
cd ../../dissertation-ai
pnpm install

Step 2: Make It Yours

In the R package — edit DESCRIPTION:

Package: your-dissertation
Title: Your Dissertation Title Here
Description: Brief description of your research
Author: Your Name

In the Quarto website — edit _quarto.yml:

website:
  title: "Your Dissertation Title"
  site-url: "https://your-username.github.io/dissertation/"

In the NextJS app — edit apps/web-app/src/app/layout.tsx:

title: 'Your Name | Dissertation Platform'

Step 3: Write Your First Content

Open quarto_website/content/chapters/introduction.qmd and start writing:

---
title: "Introduction"
---

## Research Context

[Your opening paragraph — what problem are you investigating?]

## Research Questions

1. [Your first research question]
2. [Your second research question]
3. [Your third research question]

## Significance

[Why does this matter?]

Don’t aim for perfection. Write a rough draft — we’ll iterate with AI in Part V.

Step 4: Render and Preview

# Render the website
cd dissertation-rpkg/dissertation/quarto_website
quarto render

# Preview locally
quarto preview

# Render the thesis PDF
quarto render content/thesis-main.qmd --to pdf

# Start the NextJS app
cd ../../../dissertation-ai
pnpm dev

Three views of the same content. Change a chapter, re-render, see updates everywhere.

Part V

Content Creation with AI

The write → render → review cycle

The Workflow


This is the core loop. You iterate on content, not infrastructure.

AI helps with writing, editing, and analysis — but you retain full editorial control.

AI-Assisted Writing Strategies

Drafting — “Help me write a literature review section on growth models in educational assessment.”

Editing — “This paragraph is too dense. Can you break it into clearer sub-points?”

Analysis — “Given this data summary, what are the key findings I should highlight?”

Structuring — “I have these five results. Help me organize them into a coherent narrative.”

Citation — “What seminal papers should I reference when discussing SGP methodology?”


The AI is a collaborator, not a ghostwriter. Your voice, your arguments, your interpretations.

Connecting Content to Data

The power of this framework: your writing is linked to live analyses.

## Results {#sec-results}

Our analysis of achievement trends reveals several patterns.
@fig-trends shows mean scale scores across years.

As shown in the data explorer (/explore), reading scores
demonstrate recovery following the 2020 disruption, with a
mean score of `​r summarizeAssessment("READING", year="2023")$mean_score`
by 2023.

The backtick-r syntax inserts live R values directly into your prose.

When you update your data or analysis code, the numbers in your text update too.

Part VI

Build and Deploy

From local to live in minutes

The Build Pipeline

Deploying to Vercel

  1. Push your dissertation-ai repo to GitHub
  2. Go to vercel.com → New Project → Import your repo
  3. Configure:
    • Root Directory: apps/web-app
    • Build Command: cd ../.. && pnpm build
    • Output Directory: apps/web-app/.next
  4. Deploy

Every git push to main triggers a new deployment automatically.

GitHub Pages for Quarto

Your Quarto website deploys via GitHub Actions (already configured):

# .github/workflows/quarto-render.yaml
# Triggers on push to main
# Renders website, thesis PDF, working paper
# Deploys to GitHub Pages

Two URLs, one content source:

  • Quarto site: https://your-username.github.io/dissertation/
  • NextJS app: https://your-dissertation.vercel.app
